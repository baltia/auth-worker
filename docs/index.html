Synology SSO,
reduced to a service binding.

A single Cloudflare Worker that wraps Synology's OAuth flow and hands typed sessions to its siblings over RPC. No SDK, no middleware — just one service binding.

View on GitHub
01 / Overview

A quiet auth seam for sibling workers.

Context

Synology's DSM can act as an OpenID Connect provider, but wiring it into every Cloudflare Worker you deploy is tedious. ssso-auth centralises the OAuth2 + PKCE handshake into one worker and exposes a typed RPC surface (AuthService) that sibling workers consume via Service Bindings.

Sessions live in KV. Login mode — cookie or exchange — is picked automatically based on the consumer's origin, so Pages previews and localhost work without configuration gymnastics.

02 / Capabilities

Three pieces, nothing else.

01

OAuth + PKCE

Full Synology DSM OIDC flow — login, callback, logout — implemented with arctic. No secrets leak beyond the worker.

02

Typed RPC

Consumers import AuthService types directly from the package and call methods across the binding as if they were local.

03

KV Sessions

Sessions, OAuth state, and single-use exchange codes all live in Workers KV — TTLs handled, no external database to run.

03 / Consume

Bind the service, import the types.

In the consumer worker
wrangler.jsonc 
"services": [
  { "binding": "AUTH", "service": "ssso-auth", "entrypoint": "AuthService" }
]
package.json 
"dependencies": {
  "ssso-auth": "github:mastermakrela/ssso-auth-worker"
}
fetch handler · ts 
import type { AuthService, SessionData } from "ssso-auth";

export default {
  async fetch(request: Request, env: Env) {
    const sessionId = parseCookie(request.headers.get("Cookie") ?? "", "session_id");
    if (!sessionId) return Response.redirect(env.AUTH.getLoginUrl(request.url), 302);

    const result = await env.AUTH.validateSession(sessionId);
    if (!result.valid) return Response.redirect(env.AUTH.getLoginUrl(request.url), 302);

    // ...handle authenticated request with result.session
  },
};
04 / Modes

Two login flows, picked automatically.

By consumer origin

Cookies can only be written for the auth worker's parent domain. When the consumer sits outside that scope, ssso-auth hands it a single-use code it can trade for a session over RPC.

same cookie scope as AUTH_ORIGIN cookie
*.pages.dev exchange
localhost · 127.0.0.1 · [::1] exchange
any other off-domain origin exchange
05 / RPC

The AuthService surface.

Public methods
validateSession(sessionId, requiredGroups?) → SessionValidationResult
Verifies the session and optionally checks group membership (OR logic — user needs at least one).
getLoginUrl(redirect?) → string
Builds a login URL. redirect must match the allowlist, otherwise it falls back to the auth homepage.
exchangeAuthCode(code) → { sessionId, expiresInSeconds } | null
Trades a single-use code for a session id. Codes expire after 60 seconds.
deleteSession(sessionId) → void
Server-side logout. Drops the session from KV.
06 / Setup

Deploy, then wire it up.

After Deploy-to-Cloudflare

  1. Register an OIDC client in DSM

    SSO Server → OIDC. Redirect URI = <AUTH_ORIGIN>/callback. Note the client ID and secret.

  2. Fill the vars

    In the worker dashboard (or wrangler.jsonc if cloned manually): BASE_URL, CLIENT_ID, AUTH_ORIGIN, optionally EXTRA_REDIRECT_SUFFIXES.

  3. Set the secret

    bunx wrangler secret put CLIENT_PASSWORD — the Synology OAuth client secret.

  4. Attach a custom domain

    Point the worker at AUTH_ORIGIN. Sibling workers inside that cookie scope now get cookie-mode sessions for free.
